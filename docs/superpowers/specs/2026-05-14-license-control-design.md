# Dimkey 许可证控制设计

> Status: Draft · 2026-05-14
> Owner: cube1
> 范围：Dimkey 桌面端（中文版 + 英文版）+ 营销/激活后端

## 1. 目标与范围

为 Dimkey 引入许可证控制，覆盖：

- 试用期管理（30 天软拦截 + 水印）
- 在线激活码 + Ed25519 离线签名证书
- 多设备绑定（personal 计划：3 台）
- 中英文版**一证两用**
- 海外（Lemon Squeezy）+ 国内（v1 手动 / v2 微信支付）双渠道收款
- Typora 风格的激活体验
- 客服侧最小化运营工具

**不在范围**：账户系统、订阅制、in-app 支付、in-app 退款、企业团队席位（v1 全部不做，留扩展位）。

## 2. 商业 / 产品决策（最终）

| 决策项 | 值 | 备注 |
|---|---|---|
| 商业模式 | 试用 + 付费永久授权 | 一次付费，永久使用 |
| 试用期 | 30 天 | 共享中英版，按设备 + license 维度计算 |
| 到期行为 | 软拦截 + 水印 | 不锁死功能，导出文件加一行水印 |
| 激活 | 在线激活码（首次联网） | 之后离线 14 天宽限 |
| 设备绑定 | 最多 3 台 | 用户可自助解绑释放配额 |
| 中英关系 | 通用许可证 | 同一 license 在 zh/en 版均可激活 |
| 销售渠道 | 海外 LS + 国内手动（v1）| 国内 v2 接入微信支付 |
| 定价（暂定） | $29 / ¥199 | 一次性，永久 |
| 客服 v1 | curl 脚本 + Notion 表格 | 不做 admin UI |

## 3. 试用期机制

### 3.1 试用长度
30 天。中英版**共享同一份试用记录**，安装第二个语言版本时接续。

### 3.2 三处冗余存储（防绕过 + 防回拨）
首次启动写入 3 处，每处都存 `{ first_run_at: <UTC ISO>, machine_fp: <hash>, last_run_at: <UTC ISO>, version: 1 }`：

1. **应用配置目录**
   - macOS: `~/Library/Application Support/com.dimkey/trial.json`
   - Windows: `%APPDATA%/com.dimkey/trial.json`
2. **用户主目录隐藏文件**
   - macOS: `~/.dimkey_state`
   - Windows: `%USERPROFILE%\.dimkey_state`
3. **系统凭据存储**
   - macOS: Keychain item `com.dimkey.trial`
   - Windows: Credential Manager `com.dimkey.trial`
   - 用 Rust `keyring = "3"` crate 跨平台抽象

判定规则：
- 取**最早的** `first_run_at` 作为试用起点
- 当前系统时间 < `last_run_at` → 认定时间被回拨，**用 last_run_at 继续推进计时**，置内部标志 `clock_tampered = true`（不弹窗，不报错）
- 每次启动结束后用 `now()` 更新 `last_run_at` 到 3 处

### 3.3 到期行为：软拦截 + 水印
试用到期后**应用仍可用**，但：
- 启动后顶部出现可关闭的横幅：`⚠ 试用已结束 · 导出文件将带水印`，按钮 `[ 输入许可证 ] [ 购买 ] [ 我知道了 ]`
- 所有导出文件统一加水印，水印实现位置在 `desensitizer/` 导出环节，统一调 `add_trial_watermark_if_needed()`：
  - **xlsx**：在 `Sheet1` A1 上方插一行 `本文件由 Dimkey 试用版生成 · https://dimkey.app`
  - **docx**：文档首段插入相同文字
  - **csv**：第一行 `# Dimkey 试用版 · https://dimkey.app`
  - **pdf**：第一页页脚灰色小字
  - **txt**：开头一行 `# Dimkey 试用版 · https://dimkey.app`
  - 英文版水印文案：`Generated by Dimkey trial · https://dimkey.app`

### 3.4 试用倒计时 UI
- 剩余 > 7 天：UI 不显示倒计时（在"设置 → 关于"面板能看到）
- 剩余 ≤ 7 天：设置图标旁出现橙色小角标
- 剩余 ≤ 3 天：变红色
- 不弹窗、不打扰

## 4. License 数据模型

### 4.1 License Key 格式（面向用户）
```
DK-XXXXX-XXXXX-XXXXX-XXXXX-XXXXX
```
- 字母表：`ABCDEFGHJKMNPQRSTUVWXYZ23456789`（去掉 `0/O/1/I/L`）
- 5 段 × 5 字符 = 25 char，熵 ≈ 123 bit
- 客户端输入框：自动大写 + 自动加连字符 + 粘贴时清洗空格/小写

### 4.2 设备指纹算法 v1
```
fingerprint = sha256(
    machine_id ||           # macOS IOPlatformUUID / Win MachineGuid
    primary_mac_address ||  # 字典序最小的物理网卡 MAC（排除 lo/虚拟）
    cpu_brand_string ||     # sysctl machdep.cpu.brand_string / Win32_Processor.Name
    os_install_id           # macOS IOPlatformSerialNumber / Win SID
)[0..32]   # 截 128 bit hex
```

- 任一字段读不到 → 用占位 `"unknown"`，不失败
- 算法版本常量 `FINGERPRINT_VERSION = "v1"`，未来轮换走 `v2`
- 指纹 hex 字符串展示给用户（关于面板"设备指纹 fp:a3f9...c211 📋"）供客服查证

### 4.3 本地证书 `.lic` 文件结构
落盘到 `app_config_dir/license.lic`：

```json
{
  "v": 1,
  "payload_b64": "<base64 of payload JSON>",
  "sig_b64": "<base64 of Ed25519(payload_bytes)>"
}
```

payload 解码后：
```json
{
  "license_id": "uuid",
  "license_key": "DK-XXXXX-XXXXX-XXXXX-XXXXX-XXXXX",
  "email": "user@example.com",
  "plan": "personal",
  "device_id": "uuid",
  "fingerprint": "<sha256 hex>",
  "issued_at": "2026-05-14T10:00:00Z",
  "expires_at": null,
  "next_check_at": "2026-05-21T10:00:00Z",
  "max_grace_until": "2026-05-28T10:00:00Z",
  "key_version": 1
}
```

### 4.4 客户端 LicenseState 状态机
```rust
enum LicenseState {
    Trial { days_remaining: u32 },
    TrialExpired,
    Activated { email, plan, max_devices, active_devices },
    GraceMode { email, days_until_block },   // 复验失败但还在宽限期
    Revoked { reason },                      // 后端返回 revoked → 硬拦截
    Unknown,
}
```

状态迁移：
```
未激活/Trial --输入 key--> Activating
Activating --成功--> Activated
Activating --失败--> 保留 Trial（不影响倒计时）
Activated --heartbeat 收到 revoked--> Revoked（硬拦截）
Activated --用户解绑--> 未激活（不回退 trial 倒计时）
Activated --14 天未复验成功--> GraceMode --继续--> 持续显示横幅但不锁死
Revoked --输入新 key 成功--> Activated
```

### 4.5 启动时验证流程
1. 读 `license.lic`，不存在 → 走 trial 分支（4.6）
2. 解 payload，Ed25519 验签（公钥编译期烧入）→ 失败：删 .lic，走 trial 分支
3. 算当前机器指纹，对比 payload.fingerprint → 不匹配：保持 .lic 不删，**回退到 trial 分支判定 LicenseState（Trial / TrialExpired）**，About 面板顶部额外显示一条提示 "此授权文件属于另一台机器（fp:xxxx），点此重新激活本机"（点击打开激活对话框）
4. 比较系统时间与 `next_check_at`：
   - 未到 → 直接 Activated
   - 已到 → 启动后台 heartbeat（非阻塞 UI）
5. heartbeat 成功 → 更新 next_check_at（推 +7 天）；失败但 now < max_grace_until → GraceMode；失败且超过 max_grace_until → 仍 GraceMode + 顶部黄色横幅（非锁死）；收到 `revoked` → Revoked

### 4.6 trial 分支
按 §3 三处冗余存储读取首次运行时间，算剩余天数。

### 4.7 Ed25519 密钥管理
- 私钥：Cloudflare Workers Secrets，永远不出 Workers 沙箱
- 公钥：作为 `&[u8; 32]` 常量编译进 Rust 二进制
- 轮换预案：客户端内置 `PUBKEY_V1` + `PUBKEY_V2`，证书 payload 含 `key_version` 字段，按版本选公钥验签

### 5.3 D1 Schema 完整定义

```sql
CREATE TABLE licenses (
  license_id    TEXT PRIMARY KEY,
  license_key   TEXT NOT NULL UNIQUE,
  email         TEXT NOT NULL,
  plan          TEXT NOT NULL DEFAULT 'personal',
  max_devices   INTEGER NOT NULL DEFAULT 3,
  status        TEXT NOT NULL DEFAULT 'active',  -- active|refunded|revoked|expired
  issued_at     INTEGER NOT NULL,
  expires_at    INTEGER,                          -- NULL = 永久
  source        TEXT NOT NULL,                    -- lemonsqueezy|manual_cn|wechat
  order_ref     TEXT,
  notes         TEXT,
  created_at    INTEGER NOT NULL,
  updated_at    INTEGER NOT NULL
);
CREATE INDEX idx_licenses_email ON licenses(email);
CREATE UNIQUE INDEX idx_licenses_order ON licenses(source, order_ref);

CREATE TABLE devices (
  device_id        TEXT PRIMARY KEY,
  license_id       TEXT NOT NULL REFERENCES licenses(license_id),
  fingerprint      TEXT NOT NULL,
  machine_label    TEXT,
  os               TEXT NOT NULL,
  flavor           TEXT NOT NULL,                 -- zh|en
  app_version      TEXT,
  first_activated  INTEGER NOT NULL,
  last_seen        INTEGER NOT NULL,
  deactivated_at   INTEGER                        -- NULL = 仍激活
);
CREATE INDEX idx_devices_license ON devices(license_id);
CREATE UNIQUE INDEX idx_devices_active_fp
  ON devices(license_id, fingerprint)
  WHERE deactivated_at IS NULL;

CREATE TABLE audit_log (
  id           INTEGER PRIMARY KEY AUTOINCREMENT,
  license_id   TEXT,
  device_id    TEXT,
  event        TEXT NOT NULL,
  ip           TEXT,
  user_agent   TEXT,
  detail       TEXT,    -- JSON
  ts           INTEGER NOT NULL
);
CREATE INDEX idx_audit_license ON audit_log(license_id, ts);

CREATE TABLE orders_inbox (
  source        TEXT NOT NULL,
  external_id   TEXT NOT NULL,
  payload_json  TEXT NOT NULL,
  processed_at  INTEGER,
  error         TEXT,
  PRIMARY KEY (source, external_id)
);
```

`devices` 中 `max_devices` 配额计算：`SELECT COUNT(*) FROM devices WHERE license_id = ? AND deactivated_at IS NULL`。

### 5.4 客户端 API 契约

统一响应格式：
```json
{ "ok": true,  "data": {...} }
{ "ok": false, "code": "DEVICE_LIMIT_REACHED", "message": "...", "data": {...} }
```

业务错误统一返回 HTTP 200 + `{ok:false}`，HTTP 4xx/5xx 仅用于协议/服务层错误。

#### POST /api/v1/activate
请求：
```json
{
  "license_key": "DK-XXXXX-...",
  "email": "user@example.com",
  "fingerprint": "<sha256 hex>",
  "machine_label": "Tan 的 MacBook Pro",
  "os": "macos",
  "flavor": "zh",
  "app_version": "0.7.0"
}
```
成功响应：
```json
{
  "ok": true,
  "data": {
    "license_certificate": { "v": 1, "payload_b64": "...", "sig_b64": "..." },
    "device_summary": { "current_device_id": "uuid", "active_count": 2, "max_devices": 3 }
  }
}
```

幂等：同 `(license_key, fingerprint)` 重复调用返回相同 device_id + 新签证书。

校验顺序：
1. license_key 存在 + status=active + email 一致（不一致 → `INVALID_LICENSE` 统一码，防枚举）
2. 该指纹已存在且未解绑 → 重签证书返回（幂等）
3. 当前激活设备数 ≥ max_devices → `DEVICE_LIMIT_REACHED`，在 `data.devices` 中返回设备列表给 UI 展示
4. 否则插入 device 行 + 签证书 + 返回

#### POST /api/v1/deactivate
请求：
```json
{
  "license_key": "DK-...",
  "email": "...",
  "device_id": "uuid"   // 可选；省略则按 fingerprint 解绑当前机器
}
```
不删行，置 `deactivated_at = now`，立即释放配额。

#### POST /api/v1/heartbeat
请求：`{ "license_id", "device_id", "fingerprint" }`
响应：
```json
{ "ok": true, "data": { "status": "active", "next_check_at": 1763404800000 } }
```
- 更新 `devices.last_seen`
- license.status != active 或 device.deactivated_at != null → 返回 `status: "revoked"`，客户端进 Revoked 态

#### POST /api/v1/devices/list
请求：`{ "license_key", "email" }`
响应：
```json
{
  "ok": true,
  "data": {
    "devices": [
      { "device_id": "...", "machine_label": "...", "os": "macos",
        "first_activated": 1763404800000, "last_seen": 1763491200000, "is_current": true }
    ],
    "max_devices": 3
  }
}
```

#### POST /api/v1/recover
请求：`{ "email": "user@example.com" }`
响应永远是：`{ "ok": true, "data": { "message": "如该邮箱有授权，已发送邮件" } }`（防扫号，不暴露邮箱是否存在）。后台异步发邮件。

#### POST /webhook/lemonsqueezy
- 验签 `X-Signature` (HMAC-SHA256, key=`LS_WEBHOOK_SECRET`)
- 写入 `orders_inbox`（PRIMARY KEY 去重）→ 立即返回 200
- `ctx.waitUntil(processEvent(...))` 异步处理
- 处理 `order_created` / `order_refunded`

#### Admin 接口（v1 全部 curl，无 UI）
认证：`Authorization: Bearer $ADMIN_TOKEN`。所有调用写 audit_log（detail 含 admin_token 末 4 位）。

| Endpoint | 用途 |
|---|---|
| POST /admin/issue | 手动发证（国内收款 / 售后补发） |
| POST /admin/revoke | 吊销 license |
| POST /admin/transfer | 改 license 绑定邮箱 |
| POST /admin/lookup | 按 email/key/order_ref 查 license + devices |
| POST /admin/deactivate-device | 强制解绑某设备 |
| POST /admin/resend-email | 重发激活邮件 |
| POST /admin/order-status | 查 orders_inbox 状态 |
| POST /admin/reprocess-order | 重跑失败 webhook |
| POST /admin/extend-trial | 按 fingerprint 延长试用期（特批） |

### 5.5 限流策略
基于 CF Workers Rate Limiting binding：
- `/api/v1/activate`：同 IP 10 次/小时
- `/api/v1/recover`：同邮箱 3 次/小时
- `/api/v1/heartbeat`：同 device_id 不限（客户端节流）

### 5.6 License Key 生成
```typescript
function generateLicenseKey(): string {
  const ALPHA = "ABCDEFGHJKMNPQRSTUVWXYZ23456789";
  const buf = crypto.getRandomValues(new Uint8Array(25));
  const chars = Array.from(buf, b => ALPHA[b % 31]).join("");
  return "DK-" + chars.match(/.{5}/g)!.join("-");
}
```
冲突概率 ≈ 31^25 ≈ 2.1×10^37，UNIQUE 约束兜底，碰撞重试一次。

## 8. 支付回调集成

### 8.1 Lemon Squeezy 配置
- Store: `Dimkey`
- Product: `Dimkey - Personal License`，Variant: One-time `$29 USD`
- 关闭 LS 内置 License Keys 功能（用我们自己的发证）
- **Custom Field**：在 LS Checkout 加一个隐藏字段 `lang`，由购买页 `dimkey.app/buy` 根据用户浏览器语言自动注入（zh / en）
- Webhook URL: `https://dimkey.app/webhook/lemonsqueezy`，Secret 写到 CF Secrets
- 订阅事件：`order_created`, `order_refunded`, `subscription_*`（v1 不用 sub）

### 8.2 Webhook 处理流程
```typescript
export async function onRequestPost({ request, env }) {
  const rawBody = await request.text();
  if (!verifyHmacSha256(rawBody, request.headers.get("X-Signature"), env.LS_WEBHOOK_SECRET)) {
    return new Response("invalid signature", { status: 401 });
  }
  const evt = JSON.parse(rawBody);
  const inserted = await env.DB.prepare(
    `INSERT OR IGNORE INTO orders_inbox (source, external_id, payload_json) VALUES (?, ?, ?)`
  ).bind("lemonsqueezy", evt.meta.event_id, rawBody).run();
  if (!inserted.changes) return new Response("duplicate", { status: 200 });
  ctx.waitUntil(processLemonSqueezyEvent(evt, env));
  return new Response("ok");
}
```

事件分发：
- `order_created` → `issueLicense({ email, source: "lemonsqueezy", order_ref: data.id, plan: "personal" })`
- `order_refunded` → `revokeLicenseByOrder("lemonsqueezy", data.id)`

### 8.3 issueLicense 共用函数
1. 按 `(source, order_ref)` 查 license → 已存在则重发邮件，不重复发证
2. 否则生成 license_key + 写 D1
3. 调 Resend 发邮件
4. 写 audit_log

### 8.4 邮件模板
按 LS 收单时 Custom Field `lang` 选择中/英模板。Resend 月免费额度 3000 封够用。

### 8.5 国内 v1：手动发证
- 销售形式：旺道/提贴状付店铺 / 微信群 / 微信号
- 客服 SOP：
  ```bash
  curl -X POST https://dimkey.app/admin/issue \
    -H "Authorization: Bearer $ADMIN_TOKEN" \
    -d '{"email":"u@example.com","source":"manual_cn","order_ref":"WX-20260514-001","plan":"personal","lang":"zh"}'
  ```
- order_ref 前缀约定：`WX-` 微信 / `ZD-` 旺道 / `TM-` 提贴状付
- 客服日常记录用 Notion / 飞书表格

### 8.6 国内 v2：微信支付（占位）
留好 webhook 路径 `/webhook/wechat` 与 schema `source = 'wechat'`，等微信支付商户号申请下来再接入。

### 8.7 购买页 `dimkey.app/buy`
- "Buy with USD ($29)" → LS lemon.js 弹窗
- "微信购买（¥199）" → v1 跳客服微信二维码 / v2 跳微信支付页

### 8.8 退款后行为


## 9. 错误恢复、运营与安全

### 9.1 错误码表

| code | HTTP | 用户文案 | 用户操作 |
|---|---|---|---|
| `INVALID_LICENSE` | 200 | 邮箱或许可证不正确 | 检查或找回 |
| `LICENSE_REVOKED` | 200 | 本许可证已失效（订单已退款） | 联系客服 / 重新购买 |
| `LICENSE_EXPIRED` | 200 | 许可证已到期 | 续费（v1 不会触发） |
| `DEVICE_LIMIT_REACHED` | 200 | 已达 3 台设备上限 + 设备列表 | 解绑其他设备 |
| `DEVICE_NOT_FOUND` | 200 | 设备不存在或已解绑 | — |
| `FINGERPRINT_MISMATCH` | 200 | 该证书绑定到另一台机器 | 重新激活本机 |
| `SIGNATURE_INVALID` | 200 | 授权文件已损坏，请重新激活 | 重新激活 |
| `RATE_LIMITED` | 429 | 请求过于频繁，请稍后再试 | 等待 |
| `NETWORK_UNAVAILABLE` | — | 网络不可用 | 检查网络 |
| `SERVER_ERROR` | 5xx | 服务暂时不可用，请稍后再试 | 等待 |
| `EMAIL_FORMAT_INVALID` | 200 | 请输入有效邮箱 | 修改 |
| `KEY_FORMAT_INVALID` | 200 | 许可证格式不正确 | 修改 |

### 9.2 设备迁移场景
- 主动换机：旧机器解绑 + 新机器激活，无需客服
- 旧机器损坏：`dimkey.app/account?key=...&email=...` 自助解绑；或新机器激活失败页直接看到设备列表点解绑
- 滥用检测：单 license 30 天内 deactivate+reactivate ≥10 次 → 标红 audit_log，**不自动 revoke**，人工审查

### 9.3 客服 SOP（Q1-Q4）
- Q1 "许可证被盗用"：lookup → 让用户给本机指纹 → 对比 → deactivate-device 或 revoke
- Q2 "没收到邮件"：lookup → resend-email；若 lookup 无果 → order-status → reprocess-order
- Q3 "填错邮箱"：transfer 改邮箱 + 自动重发到新邮箱
- Q4 "退款"：LS 用户自助退；国内手动渠道用 admin/revoke

### 9.4 灾难恢复预案

| 故障 | 应对 |
|---|---|
| CF Workers 全球宕机 | 客户端宽限期 14 天兜底；横幅提示"暂时无法连接服务器" |
| D1 损坏 / 误删 | CF D1 每日快照保留 30 天；orders_inbox 留底，webhook 可重放 |
| Ed25519 私钥泄露 | 启动密钥轮换：新版客户端内置 pubkey_v2 → 后端切签新证书 v2 → 老 v1 证书 14 天后失效 |
| 客户端 .lic 被删 | 重新激活：指纹不变 → 后端识别已存在设备 → 重签证书（幂等） |
| 系统时间回拨 | 三处冗余 + last_run_at 检查（§3.2） |
| webhook 重试风暴 | UNIQUE(source, external_id) 兜底 |
| 邮件投递失败 | Resend 重试 3 次后写 audit_log；客服 Q2 流程处理 |

### 9.5 监控
- CF Workers Analytics 自带
- 每日 cron Worker 查 audit_log 汇总：今日新激活 / 退款 / webhook 失败 / 异常设备迁移，邮件发送给运营邮箱

### 9.6 安全审计清单
- [x] Ed25519 私钥只在 Workers Secrets，不进 git
- [x] webhook secret HMAC 校验
- [x] admin token ≥ 32 字符
- [x] 邮箱枚举防护：recover 不暴露存在性、错误码统一 `INVALID_LICENSE`
- [x] HTTPS-only
- [x] 客户端公钥编译期烧入
- [x] 客户端不发送文件内容
- [x] 限流（按 IP + 按邮箱双维度）
- [x] audit_log 不记 license_key 全文（只记末 4 字符 + license_id）

### 9.7 隐私与合规
- 客户端发给后端的字段（明示在 `dimkey.app/privacy`）：
  - email, license_key, fingerprint, machine_label, os, flavor, app_version
  - 自动：ip, user_agent
- **不发送**任何文件内容、识别结果、扫描数据
- 设备指纹是 sha256 哈希后的 hex，不可逆推 MAC / serial
- 数据保留：audit_log 1 年；devices 永久至 deactivated_at + 90 天清理
